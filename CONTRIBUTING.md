# Contributing to the Axios API Handbook

First off, thank you for considering contributing to the Axios API Handbook! It's people like you who make this a valuable resource for the developer community.

To maintain the high quality and professional standards of this repository, we ask that you follow these guidelines.

## How Can I Contribute?

### Adding a New API Guide
If you'd like to add a new API to the `fake-apis/` directory, please ensure it follows this structure:
1.  **Title**: Clear name of the API.
2.  **Technical Details**: Description, Base URL, Authentication requirements, and Encryption status.
3.  **Key Endpoints**: A table showing the most useful endpoints and what they return.
4.  **Implementation Example**: A clean, tested Axios code snippet.
5.  **Sample Response**: A JSON block showing the data structure.
6.  **Use Cases**: 3-4 practical ways to use the API.

### Improving Documentation
If you find a typo, a dead link, or an outdated implementation, please open a Pull Request.

## Style Guidelines

### Professional Tone
This handbook follows a "humanized" and professional tone. Avoid slang and keep explanations concise and technical yet accessible.

### No Emojis
To maintain a clean and professional look, **do not use emojis** in the documentation or code comments.

### JavaScript Standards
*   Use ES6+ syntax (Arrow functions, Destructuring, etc.).
*   Always use `async/await` for asynchronous operations in examples.
*   Include basic error handling (`try/catch`) in all code snippets.

## Pull Request Process

1.  **Fork the repo** and create your branch from `main`.
2.  **Test your code**: If you are adding a code example, run it locally to ensure it works.
3.  **Update the README**: If you add a new API, make sure to add it to the list in `README.md`.
4.  **Submit the PR**: Fill out the Pull Request template completely.
5.  **Review**: At least one maintainer must review and approve your PR before it is merged.
